================================================================================
DROPLET DEPLOYMENT — webshop screenshot collector
Written 2026-06-09. Mirrors the claude_birdeye_trending / claude_funda Docker
setup: one long-running container on a DigitalOcean droplet. Replaces the old
screenshot-watchdog.sh + systemd unit approach.
================================================================================

WHAT RUNS
---------
One long-running container. Its main process is collect.py, which every
POLL_HOURS (default 1h) runs one screenshot cycle and then sleeps. A cycle:
  1. loads the shop registry from JSON_URL (shop_registry.external.json: a dict
     with a "shops" object keyed by name -> {url, category, resolved_on}),
  2. lists the R2 "screenshots" bucket to find each shop's most recent capture,
  3. for every shop NOT captured in the last DEDUPE_HOURS (default 24h):
     launches a FRESH headless Chromium, opens the page, dismisses cookie
     banners, screenshots it, uploads the JPEG to R2, then quits Chromium.
A failed shop is logged and the cycle continues; a failed cycle is logged and
the loop continues. Docker's restart policy is the backstop if the process dies.

WHY THIS REPLACES THE WATCHDOG
------------------------------
The old screenshot.py was a one-shot; screenshot-watchdog.sh (a systemd service)
relaunched it ~120x/day and force-killed each container after 30 min because
Chromium leaks memory on the 1GB box. Here the script IS the scheduler (hourly
loop) and Chromium is launched+quit PER SHOP, so memory is reclaimed between
shops instead of being force-killed. The 24h-per-shop dedupe means actual
screenshot output is unchanged (~1/shop/day) — only the churn is gone.

DOWNSTREAM CONSUMER
-------------------
The "image_openai" project (container `eije2`, repo coconutsRhealthy/image_openai)
reads these screenshots from the same R2 "screenshots" bucket, compares each
shop's latest vs previous capture, and uses OpenAI to detect new promotions.
This collector is the producer; eije2 is the consumer. Don't stop this without
knowing eije2 then has no fresh images to compare.

THE FILES (on the droplet, in /root/screenshot-bot)
---------------------------------------------------
  Dockerfile          - builds the image (python:3.11-slim + chromium + driver)
  requirements.txt    - selenium, Pillow, boto3, requests
  screenshot.py       - run_cycle(): one full pass (the work)
  collect.py          - the hourly scheduler loop (container main process)
  .env                - secrets, NOT in git (see CONFIG below)

DEPLOY COMMANDS
---------------
On the Mac, from the project dir, copy the files up:
  ssh root@DROPLET_IP 'mkdir -p ~/screenshot-bot'
  scp Dockerfile requirements.txt screenshot.py collect.py \
      root@DROPLET_IP:~/screenshot-bot/

Create /root/screenshot-bot/.env on the droplet (once; keep it off git):
  R2_ACCOUNT_ID=...
  R2_ACCESS_KEY_ID=...
  R2_SECRET_ACCESS_KEY=...
  # optional overrides: R2_BUCKET_NAME, R2_PREFIX, JSON_URL, POLL_HOURS, DEDUPE_HOURS

On the droplet, in /root/screenshot-bot:
  docker build -t screenshot-bot .

  # smoke test — one cycle, then it would loop; Ctrl+C after you see uploads.
  # (or override DEDUPE_HOURS=0 to force a re-capture of one shop)
  docker run --rm --env-file .env screenshot-bot python screenshot.py

  # start the collector (this command IS the deployment):
  #   --init is REQUIRED: it runs tini as PID 1 to reap Chrome's zombie
  #   subprocesses. Without it, each Chrome launch leaves ~11 unreaped zombies
  #   (python as PID 1 does not reap); they hit the container task limit (~1024)
  #   in ~9h and ALL Chrome launches then fail with "Chrome instance exited".
  #   (Learned the hard way 2026-06-10 — see INCIDENT note at bottom.)
  #   --cpus="0.7" stops Chromium pinning the single core (the value the old
  #   watchdog used). Memory is left unbounded on purpose: swap (see below) is
  #   the safety net; a hard --memory cap risks Chromium being OOM-killed inside
  #   the container on a heavy page.
  docker run -d --name screenshot-bot --restart unless-stopped \
      --init --cpus="0.7" --env-file .env screenshot-bot

(Replace DROPLET_IP with the droplet's address.)

CONFIG (env vars)
-----------------
  R2_ACCOUNT_ID          (required) Cloudflare R2 account id
  R2_ACCESS_KEY_ID       (required)
  R2_SECRET_ACCESS_KEY   (required)
  R2_BUCKET_NAME=screenshots   target bucket
  R2_PREFIX=                    key prefix in the bucket
  JSON_URL=...                  shop registry (defaults to shop_registry.external.json in R2)
  POLL_HOURS=1                  cycle cadence in hours
  DEDUPE_HOURS=24               skip a shop captured within this many hours
  SHOP_TIMEOUT_SECONDS=90       hard wall-clock cap per shop; over this the whole
                                Chromium process tree is force-killed (see INCIDENT 2026-06-11)
  PAGE_LOAD_TIMEOUT_SECONDS=45  abort a single page load (Selenium default ~300s)
  CLEANUP_TIMEOUT_SECONDS=15    cap on graceful driver.quit() before force-kill
  SKIP_SHOPS=                   comma-separated shop keys to skip; ADDED to the built-in
                                blocklist (currently {meetmethere}, a chronic hanger)
  BLOCKLIST_RETRY_HOURS=24      blocklisted shops are not skipped forever — each gets one
                                real "probation" attempt this often (the 90s cap makes it
                                safe). A probe that succeeds lands in R2 normally, so a
                                fixed shop recovers on its own; a still-broken shop times
                                out, uploads nothing, and waits another interval. The last
                                attempt time is persisted in R2 under _probation/<shop>.txt

MONITORING
----------
  docker ps                              - screenshot-bot should show "Up"
  docker logs --tail 30 screenshot-bot   - recent activity
  docker logs -f screenshot-bot          - live tail
Each cycle logs: "ok  cycle done  shot:N skipped:M failed:K".

OPERATIONS
----------
  docker stop screenshot-bot    - pause collection
  docker start screenshot-bot   - resume
  free -m                       - check droplet memory headroom

REDEPLOY AFTER A CODE EDIT (no persistent data to preserve — output lives in R2)
--------------------------------------------------------------------------------
  # re-scp the changed file(s) from the Mac, then on the droplet:
  cd ~/screenshot-bot
  docker build -t screenshot-bot .
  docker rm -f screenshot-bot
  docker run -d --name screenshot-bot --restart unless-stopped \
      --init --cpus="0.7" --env-file .env screenshot-bot

HOST: SWAP (the 1 GB / 1 vCPU box)
----------------------------------
The droplet has 1 GB RAM and no swap by default — the root cause of the old
OOM fragility. A 2 GB swapfile was added 2026-06-09 as the memory safety net:
  fallocate -l 2G /swapfile && chmod 600 /swapfile
  mkswap /swapfile && swapon /swapfile
  echo "/swapfile none swap sw 0 0" >> /etc/fstab          # persist across reboot
  sysctl -w vm.swappiness=10
  echo "vm.swappiness=10" > /etc/sysctl.d/99-swappiness.conf
Verify with `swapon --show` and `free -m`. swappiness=10 means the box only
leans on swap under real RAM pressure, not while RAM is still free.

IF SCREENSHOTS START FAILING
----------------------------
Logs show "Error on <shop>: ...". A single shop failing (slow page, layout
change, cookie wall) is logged and skipped; the next cycle retries it. If EVERY
shop fails, check: R2 credentials (.env), JSON_URL reachability, and that
chromium/chromedriver still launch (docker logs will show a webdriver error).
The loop keeps running through failures; one missed cycle is not a problem.

INCIDENT 2026-06-10 — zombie PID exhaustion ("Chrome instance exited")
---------------------------------------------------------------------
Symptom: after ~9h of healthy cycles the bot logged shot:0 with every shop
failing "session not created: Chrome instance exited", while the container
stayed "Up" (0 restarts). NOT memory/OOM — the container used only ~138 MB.
Cause: the container was started WITHOUT --init, so python was PID 1 and did
not reap Chrome's orphaned subprocesses. Each screenshot left ~11 zombies
(chromium + chrome_crashpad); after ~98 screenshots the container held ~1100
zombie tasks and hit the per-container task ceiling, so fork() for any new
Chrome failed.
Fix: recreate WITH --init (tini as PID 1 reaps zombies). Verify after the fix:
  docker exec screenshot-bot ps -p 1 -o comm=        # -> docker-init (not python)
  docker exec screenshot-bot ps -e -o stat= | grep -c Z   # -> stays ~0, not climbing
Lesson: any container that launches Chrome per-iteration MUST run with --init.

INCIDENT 2026-06-11 — single-shop hang pegging CPU for ~9h (no wall-clock cap)
-----------------------------------------------------------------------------
Symptom: DO dashboard showed sustained ~70% CPU for ~10h. screenshot-bot stuck —
last log "Opening meetmethere -> https://meet-me-there.com/?country=NL" at
2026-06-10 15:39, no progress for ~9h13m. One chromium pid had burned ~6h of CPU
at 63% busy-looping; container at 81% CPU / 133 PIDs. Zombies were 0 (so NOT the
--init bug above) — this was a single LIVE chromium spinning. (Flagged by the
eije2/image_openai Claude instance on the same droplet.)
Cause: meet-me-there.com finishes loading, then spins Chromium in a busy-loop.
run_cycle() had NO hard wall-clock cap on a shop — Selenium's renderer/page-load
timeout doesn't cover this post-load phase — so screenshot_one() blocked forever
and the cycle never advanced. driver.quit() also didn't reap the wedged tree.
Immediate fix: docker restart screenshot-bot (safe; dedups off R2, no data loss).
Durable fix (screenshot.py): (1) SIGALRM per-shop wall-clock cap
SHOP_TIMEOUT_SECONDS=90 that force-kills the whole chromedriver->chromium tree
via psutil.kill_proc_tree(); (2) set_page_load_timeout(45) so a stuck GET aborts;
(3) SKIP_SHOPS blocklist (meetmethere) — also produces blank <40KB screenshots,
bad input for eije2. NOT permanent: each blocklisted shop gets one real
"probation" attempt every BLOCKLIST_RETRY_HOURS (default 24h; last-attempt time
persisted in R2 under _probation/<shop>.txt, written before the attempt so a hang
can't cause hourly re-tries), so a shop that gets fixed recovers on its own.
Verify after the fix:
  docker logs screenshot-bot | grep -i meetmethere   # -> "Skipping ... blocklist"
  docker stats --no-stream screenshot-bot            # CPU not pinned for hours
  docker exec screenshot-bot ps -e -o comm= | grep -c chromium  # not climbing
Lesson: any per-iteration Chrome launch needs a wall-clock cap that force-kills
the process tree — a renderer/page-load timeout alone does NOT bound CPU.
